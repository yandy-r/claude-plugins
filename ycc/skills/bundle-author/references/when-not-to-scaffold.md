# When Not to Scaffold

Decision guide for avoiding unnecessary scaffolding. Before creating anything new,
check each anti-pattern below.

---

## 1. Duplicating an Existing Skill

**Rationale:** A second skill that covers the same domain fragments discovery and
confuses invocation routing.

**Do this instead:** Run `ls ycc/skills/` before scaffolding. If a skill already handles
the domain, extend it (add a phase, a flag, or a reference file) rather than creating a
new one. If the name no longer fits the expanded scope, rename the directory.

---

## 2. One-Off Task

**Rationale:** Scaffolding a skill for a task you will run once creates permanent
maintenance surface with no ongoing value.

**Do this instead:** Run the task using a direct prompt or an existing general-purpose
skill such as `ycc:ask`. Only scaffold when the same workflow will be reused repeatedly.

---

## 3. Logic That Belongs in `_shared/`

**Rationale:** Copying the same bash logic into multiple skill `scripts/` directories
creates drift and makes fixes harder to apply consistently.

**Do this instead:** If two or more existing skills need the same helper, add it to
`ycc/skills/_shared/scripts/` and source it from each skill via
`${CLAUDE_PLUGIN_ROOT}/skills/_shared/scripts/<name>.sh`. Do not create a new skill
just to house shared logic.

---

## 4. New Top-Level Plugin

**Rationale:** `ycc` is the single plugin in this repo. Adding entries to
`.claude-plugin/marketplace.json` breaks the 2.0 consolidation contract.

**Do this instead:** Add a new skill under `ycc/skills/`. The `ycc` namespace
accommodates unlimited skills without requiring a new plugin registration.

---

## 5. Generic Scaffolding Across External Repos

**Rationale:** `bundle-author` is scoped strictly to the `ycc/` source tree in this
repo. Scaffolding for arbitrary external projects is a different concern.

**Do this instead:** Use the `skill-creator` skill, which is designed for creating and
modifying skills in other projects and contexts outside this bundle.

---

## 6. Hand-Editing Generated Bundles

**Rationale:** `.cursor-plugin/` and `.codex-plugin/` are fully regenerated by
`./scripts/sync.sh`. Any manual edit is silently overwritten on the next sync.

**Do this instead:** Make all source changes under `ycc/` (skills, commands, agents,
shared scripts), then run `./scripts/sync.sh` to propagate them to the generated
targets.

---

## 7. Adding an Agent Without a Consumer

**Rationale:** An agent that no skill or command invokes is dead code — it adds weight
to the bundle and misleads future maintainers about what is active.

**Do this instead:** Identify the skill or slash command that will invoke the agent
before creating it. If no consumer exists or is planned, skip the agent entirely. You
can always add it later when the consuming skill is written.

---

Guardrails section of `CONTRIBUTING.md` (repo root) for the policy-level view.

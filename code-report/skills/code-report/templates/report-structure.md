# Report Structure Template

This template defines the structure for implementation reports generated by the code-report skill.

## Frontmatter

```yaml
---
title: [Feature Name] Implementation Report
date: [mm/dd/yyyy]
original-plan: docs/plans/[feature-name]/parallel-plan.md
---
```

## Overview

Write 2-4 sentences that provide:

- What was implemented
- High-level approach taken
- Key architectural decisions made

Keep it information-dense and focused on the "what" and "why".

Example:

```
Implemented user authentication system using JWT tokens with refresh token rotation.
The system integrates with the existing user management API and adds new middleware
for route protection. Authentication state is managed on the frontend using React
Context with automatic token refresh handling.
```

## Files Changed

List all files that were created or modified, grouped by type.

### Created

```markdown
- `/path/to/new/file.ext`: Brief description of what this file does
- `/another/new/file.ext`: Purpose and role in the feature
```

### Modified

```markdown
- `/path/to/existing/file.ext`: Brief description of changes made
- `/another/existing/file.ext`: What was modified and why
```

## New Features

List each feature with a short name and one-sentence description of what it does and how it works.

Include all features, both new and changed.

```markdown
**Feature Name**: Brief description of what it does and how it works.

**Another Feature**: Description with implementation approach.
```

Example:

```markdown
**JWT Authentication**: Token-based authentication with access and refresh tokens,
implementing automatic token rotation on the 15-minute expiry cycle.

**Protected Routes**: Middleware-based route protection that checks JWT validity
and user permissions before allowing access to protected resources.
```

## Additional Notes

Include important information that would be valuable for maintainers:

- Important implementation details
- Concerns or technical debt
- Dependencies added (packages, libraries, services)
- Configuration changes required
- Migration steps if needed
- Performance considerations
- Security considerations
- Known limitations

Be thorough but concise. Only include information that's important for someone maintaining this code.

Example:

```markdown
- Added `jsonwebtoken` and `bcrypt` dependencies to package.json
- Requires `JWT_SECRET` and `JWT_REFRESH_SECRET` environment variables to be set
- Database migration required: Run `npm run migrate:auth` to add auth tables
- Current implementation stores refresh tokens in memory - should migrate to Redis for production
- Rate limiting on auth endpoints set to 5 attempts per minute per IP
```

## E2E Tests To Perform

Provide specific testing guidance for QA. Each test should include:

- Test name/description
- Step-by-step instructions
- Expected results
- Edge cases to verify

Write this so QA knows exactly what to test and how.

```markdown
### Test 1: [Test Name]

**Steps:**

1. Step one
2. Step two
3. Step three

**Expected Result:**
Describe what should happen

**Edge Cases:**

- Edge case to test
- Another edge case
```

Example:

```markdown
### Test 1: User Login Flow

**Steps:**

1. Navigate to `/login`
2. Enter valid credentials (user: test@example.com, password: Test123!)
3. Click "Login" button
4. Verify redirect to dashboard

**Expected Result:**
User should be logged in, redirected to `/dashboard`, and see personalized welcome message with username.

**Edge Cases:**

- Try with invalid password - should show error message
- Try with unregistered email - should show "User not found"
- Try 6 times with wrong password - should trigger rate limiting

### Test 2: Token Refresh

**Steps:**

1. Log in successfully
2. Wait 16 minutes (token expires at 15 minutes)
3. Perform any action that requires authentication
4. Verify action succeeds

**Expected Result:**
Token should be automatically refreshed in the background, and the action should complete successfully without requiring re-login.

**Edge Cases:**

- Test with network disconnected during refresh - should retry
- Test after 7 days (refresh token expiry) - should require re-login
```

## Notes on Quality

### Overview Quality

- Information-dense, not fluffy
- Focus on what was done and why
- Mention key architectural decisions
- 2-4 sentences maximum

### Files Changed Quality

- Use relative paths from project root
- Brief but useful descriptions
- Group logically (created vs modified)

### Features Quality

- Clear, descriptive feature names
- One-sentence descriptions
- Include "how it works" context
- Focus on user-facing or architectural features

### Additional Notes Quality

- Only include important information
- Be honest about concerns or technical debt
- Mention required follow-up work
- List dependencies and configuration needs

### E2E Tests Quality

- Specific, actionable test cases
- Step-by-step instructions anyone can follow
- Clear expected results
- Don't forget edge cases and error scenarios
- Include both happy path and failure scenarios

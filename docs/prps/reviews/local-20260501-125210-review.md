## Summary

Six HIGH findings span the new releaser publish flow and the changelog rewrite: SKILL.md metadata is out of sync with the new `--publish` surface, the `draft-changelog.sh` rewrite re-derives `PREVIOUS_TAG` independently of the resolved `FROM_REF` (causing the compare URL and "Previous release" line to diverge from the actual commit range), and unvalidated CLI inputs flow into `git log` and a printed `gh` command string. Eleven MEDIUM and seven LOW findings round out template/script alignment, performance, and defense-in-depth gaps. No CRITICAL issues. Generated mirrors under `.codex-plugin/`, `.cursor-plugin/`, `.opencode-plugin/` were excluded — drift there is expected generator output. Internal planning artifacts under `docs/prps/plans/completed/` and `docs/prps/reports/` were also excluded.

### HIGH

- **[F001]** `ycc/skills/releaser/SKILL.md:3` — `description` field never mentions `--publish` or any publish-trigger phrase, so skill routing won't activate this skill when the user asks to "publish a release."
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Append a trigger phrase such as `"publish a GitHub release"` / `"publish release notes"` to the `description` field so the router can activate the skill for publish requests. Update the closing clause that still says "Never auto-commits, pushes, or publishes" to reflect that `publish-release.sh` is now an explicit, confirmation-gated path.

- **[F002]** `ycc/skills/releaser/SKILL.md:143` — Phase 4 fenced code block shows `draft-changelog.sh <new-version> [<from-ref>]` but the script's new CLI signature is `[options] <new-version> [<from-ref>]`; the prose immediately below correctly mentions `--exclude-internal` and `--template`, but an agent reading only the fenced block will invoke the script without those flags.
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Update the code block to `draft-changelog.sh [--exclude-internal] [--template <path>] <new-version> [<from-ref>]` so the block and the surrounding prose are consistent. Either that, or split into two examples (flag-less + flag form) matching the style `publish-release.sh`'s usage block uses.

- **[F003]** `ycc/skills/releaser/scripts/draft-changelog.sh:149` — `${GIT_RANGE}` (derived from positional arg `$2`) is passed unvalidated to `git log`; a from-ref beginning with `-` would be interpreted as a flag.
  - **Status**: Fixed
  - **Category**: Security
  - **Suggested fix**: Validate `FROM_REF` early — directly after positional parsing add: `if [[ "${FROM_REF}" == -* ]]; then echo "draft-changelog.sh: FAIL: from-ref cannot begin with '-'" >&2; exit 1; fi`. Optionally also assert the ref resolves with `git rev-parse --verify --quiet "${FROM_REF}"`.

- **[F004]** `ycc/skills/releaser/scripts/draft-changelog.sh:263` — `PREVIOUS_TAG` is computed by a second independent `git describe --tags --abbrev=0` after `FROM_REF` was already resolved (possibly to a user-supplied SHA or older tag). The compare URL and "Previous release" line then point to the latest reachable tag, not the actual range left bound — they silently diverge from the commit log being rendered.
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Capture the `git describe` result once during the `FROM_REF` resolution block (lines 108–113) into `PREVIOUS_TAG`; set `FROM_REF="${PREVIOUS_TAG}"` from it when auto-resolving, and preserve the user-supplied value when `FROM_REF` came from CLI. Remove the second `git describe` at line 263 and reuse the already-resolved `PREVIOUS_TAG`.

- **[F005]** `ycc/skills/releaser/scripts/draft-changelog.sh:263,272` — When no prior tag exists `PREVIOUS_TAG` falls back to the literal string `"(no previous tag)"`, which is then interpolated unguarded into `COMPARE_URL`, producing `https://github.com/owner/repo/compare/(no previous tag)...v1.4.0` in the rendered notes.
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Guard the `COMPARE_URL` construction: only set it when `PREVIOUS_TAG` is a real ref. Wrap line 272 in `if [[ "${PREVIOUS_TAG}" != "(no previous tag)" ]]; then COMPARE_URL=...; fi`. Leave `COMPARE_URL=""` otherwise, and (paired with **[F008]**) suppress the entire "Full Changelog" line in the template when empty.

- **[F006]** `ycc/skills/releaser/scripts/publish-release.sh:186-188` — `RESOLVED_CMD` interpolates `${TAG}` and `${NOTES_FILE}` unquoted into a string that is later printed as a "the command we will run" preview. The actual `gh` invocations at lines 211–213 are word-split safely, but the preview string is a footgun: any caller who pipes it to `eval`, `bash -c`, or copy-pastes it gets command injection from a tag like `v1.0.0" --draft "` or a notes path with shell metacharacters.
  - **Status**: Fixed
  - **Category**: Security
  - **Suggested fix**: Use `printf '%q'` when constructing `RESOLVED_CMD` so the displayed preview is a valid, safely-quotable shell command and accurately represents what will execute: `RESOLVED_CMD="gh release create $(printf '%q' "${TAG}") --notes-file $(printf '%q' "${NOTES_FILE}") --title $(printf '%q' "${TAG}")"`.

### MEDIUM

- **[F007]** `ycc/skills/releaser/SKILL.md:259-263` — Phase 8 shows a fenced reference to `publish-release.sh` but no concrete worked example with values (e.g. `publish-release.sh v1.4.0 docs/releases/v1.4.0.md --mode=auto`). The script's own header has them; SKILL.md (the surface the AI reads) does not, leaving routing for `--publish` underspecified.
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Add a brief Examples subsection under Phase 8 (or in the Arguments section) showing at least one concrete `--publish --confirm` invocation, mirroring the script's own usage block.

- **[F008]** `ycc/skills/releaser/references/release-notes-template.md:37` — When `COMPARE_URL` is empty (no `origin` remote, or guarded out by **[F005]**), substitution leaves `**Full Changelog**:` with a trailing space and no URL.
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Either add a sentinel line (e.g. wrap in an HTML comment the Python block strips when `compare_url == ""`) or have the substitution block remove the entire "Full Changelog" line when the URL is absent.

- **[F009]** `ycc/skills/releaser/scripts/draft-changelog.sh:188-208` — Breaking commits with internal types (e.g. `refactor!`, `docs!`) are added to `BREAKING_LINES` and also to `INTERNAL_LINES`, so they appear twice in the rendered notes (under `## Breaking Changes` and again inside the collapsible Maintenance block).
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: After placing a commit into `BREAKING_LINES`, skip its type bucket: `if [[ ${is_breaking} -eq 1 ]]; then BREAKING_LINES+=("${line}"); continue; fi`.

- **[F010]** `ycc/skills/releaser/scripts/draft-changelog.sh:153` — A second full `git log` traversal at line 153 (`COMMITS_ONELINE`) walks the same range already collected in `LOG_TMP` at line 149, doubling git-log I/O.
  - **Status**: Fixed
  - **Category**: Performance
  - **Suggested fix**: Build the oneline output during the existing bucketize loop (append `"${hash} ${subject}"` to an `ONELINE_LINES` array, join at the end). This eliminates the second `git log` call entirely.

- **[F011]** `ycc/skills/releaser/scripts/draft-changelog.sh:193` — The bucket pattern `revert:*|revert\(*\):*` does not cover `revert!*` or `revert\(*\)!:*`. The same breakout was added for `feat`/`fix` (lines 188–191). A `revert!: …` commit currently falls through to `bucket="other"` even though its breaking marker is detected.
  - **Status**: Fixed
  - **Category**: Pattern Compliance
  - **Suggested fix**: Extend the case branch to `fix:*|fix\(*\):*|fix!*|fix\(*\)!:*|revert:*|revert\(*\):*|revert!*|revert\(*\)!:*)`.

- **[F012]** `ycc/skills/releaser/scripts/draft-changelog.sh:248` — `ALL_INTERNAL_LINES` is concatenated using the `${arr[@]+"${arr[@]}"}` default-expansion idiom. Correct under `set -u`, but inconsistent with the surrounding `[[ ${#arr[@]} -gt 0 ]]` style used elsewhere in this script — adds cognitive load.
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Use the same conditional-append style as the rest of the script:

    ```bash
    ALL_INTERNAL_LINES=()
    [[ ${#INTERNAL_LINES[@]} -gt 0 ]] && ALL_INTERNAL_LINES+=("${INTERNAL_LINES[@]}")
    [[ ${#OTHER_LINES[@]} -gt 0 ]]    && ALL_INTERNAL_LINES+=("${OTHER_LINES[@]}")
    ```

- **[F013]** `ycc/skills/releaser/scripts/draft-changelog.sh:267` — Remote-URL normalization handles only the exact `git@github.com:` SCP pattern. `ssh://git@github.com/...`, GitHub Enterprise hosts, and GitLab SSH all pass through un-transformed and embed a malformed URL in release notes.
  - **Status**: Fixed
  - **Category**: Security
  - **Suggested fix**: Validate the result before embedding: after the substitution, check `if [[ "${http_url}" != https://* ]]; then COMPARE_URL=""; fi`. Optionally extend the rewrite to cover `ssh://git@<host>/` and `git@<host>:` for any host, not just `github.com`.

- **[F014]** `ycc/skills/releaser/scripts/draft-changelog.sh:315` — The `.replace("{{COMPARE_URL}}", compare_url)` line is misaligned by one space relative to every other `.replace(...)` call in the chain (all others use two spaces of padding after `{{...}}`).
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Change `"{{COMPARE_URL}}", compare_url` to `"{{COMPARE_URL}}",  compare_url` (two spaces) to match surrounding alignment.

- **[F015]** `ycc/skills/releaser/scripts/publish-release.sh:141` — `git rev-parse "${TAG}"` is called without `--verify --quiet` on user-supplied input, so git diagnostic output (packed-refs failures, partial-clone messages) can leak to stderr in edge cases.
  - **Status**: Fixed
  - **Category**: Security
  - **Suggested fix**: Use `git rev-parse --verify --quiet "${TAG}" >/dev/null 2>&1` (matching the pattern in `draft-changelog.sh:125`) to suppress incidental git diagnostic output.

- **[F016]** `ycc/skills/releaser/scripts/publish-release.sh:197-206` — In `--confirm` + `--mode=edit` the existing release body is shown line by line via `warn "${line}"`, which prefixes every line with `publish-release.sh: WARN:`. Multi-paragraph release bodies become a wall of prefixed text and the `warn "---"` framing is buried in the same noise.
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Print the body fence and raw body directly to stderr without the `warn` prefix (`printf '%s\n' "---" "${existing_body}" "---" >&2`), and reserve the `warn` prefix for the "about to overwrite release body for ${TAG}" summary line only.

- **[F017]** `ycc/skills/releaser/scripts/publish-release.sh:199` — `gh release view "${TAG}"` is called twice — once for existence detection (line 157) and again for the body fetch (line 199). Two round-trips where one would do; if `gh` is rate-limited the issue surfaces twice.
  - **Status**: Fixed
  - **Category**: Performance
  - **Suggested fix**: Cache the JSON at the existence-detection step: `RELEASE_JSON="$(gh release view "${TAG}" --json url,tagName,body 2>/dev/null || true)"`. Derive `RELEASE_EXISTS` and `existing_body` from it.

### LOW

- **[F018]** `ycc/skills/releaser/SKILL.md:288-292` — The "never bypasses confirmation, even when the user passes `--publish` and `--confirm` together" rule is policy text, not enforced by code. Compliance depends on the LLM following the prompt; the actual second-layer enforcement is the human re-running the script with `--confirm` as a separate shell command.
  - **Status**: Fixed
  - **Category**: Security
  - **Suggested fix**: Acknowledge the layered model in the Important Notes paragraph: name `publish-release.sh`'s `--confirm` requirement as the enforcement layer, and note that the LLM cannot forge a second invocation. No code change required.

- **[F019]** `ycc/skills/releaser/scripts/draft-changelog.sh:22` — `SCRIPT_DIR` is computed near the top but the section header above `TEMPLATE` (line 43) makes it look like a stray variable, even though `TEMPLATE` does use it.
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Move `SCRIPT_DIR` and the `TEMPLATE` default assignment under a single `# ── defaults ──` section before argument parsing so they read as a paired declaration.

- **[F020]** `ycc/skills/releaser/scripts/draft-changelog.sh:184` — Commit subjects containing parentheses, backticks, or `$(...)` flow into the rendered Markdown via `.replace()`. There is no code-execution path (Python `str.replace`, not `eval`), but a subject that begins with `-` could accidentally render as a nested list bullet.
  - **Status**: Fixed
  - **Category**: Security
  - **Suggested fix**: Strip or escape a leading `-` in raw commit subjects before formatting as a list item. No execution risk to address — purely Markdown rendering polish.

- **[F021]** `ycc/skills/releaser/scripts/draft-changelog.sh:271` — Only `git@github.com:` SSH remotes are normalized. GitLab SSH (`git@gitlab.com:`) and similar silently produce broken compare URLs with no documented limitation.
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Add a comment noting that only GitHub SSH and HTTPS remotes are normalized; everything else produces best-effort or empty output. Pair with **[F013]**'s validate-or-blank approach.

- **[F022]** `ycc/skills/releaser/scripts/draft-changelog.sh:278-290` — `COMMITS_ONELINE` is passed as `sys.argv[9]` to the inline Python; on repos with thousands of commits since the last tag this can be tens of kilobytes. Bounded by `ARG_MAX` (~2 MB on Linux) so not a hard limit, but unusual.
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Document the `ARG_MAX` ceiling near line 276's "raw oneline log" comment. If large repos become a concern, pass via a temp file and read inside the Python block with `pathlib.Path.read_text()`.

- **[F023]** `ycc/skills/releaser/scripts/publish-release.sh:157` — `gh release view "${TAG}"` is called unconditionally, even when `--mode=create` is explicit. In create mode this is a wasted network round-trip plus a suppressed "release not found" error.
  - **Status**: Fixed
  - **Category**: Performance
  - **Suggested fix**: Skip the `gh release view` call when `MODE="create"` is set explicitly: wrap in `if [[ "${MODE}" != "create" ]]; then ... fi`. The mode-conflict check at lines 163–167 can short-circuit with `RELEASE_EXISTS=0`.

- **[F024]** `ycc/skills/releaser/references/release-notes-template.md:29` — The `<details><summary>Commit Log</summary>` block has no commit count, while the Maintenance block uses `Maintenance (N commits)`. Minor inconsistency between the two collapsibles.
  - **Status**: Fixed
  - **Category**: Maintainability
  - **Suggested fix**: Add a `{{COMMITS_COUNT}}` placeholder to the Commit Log heading, or document in a comment that the Commit Log is intentionally unnumbered (raw oneline log, may be very long).

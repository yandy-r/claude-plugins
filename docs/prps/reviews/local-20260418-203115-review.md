### MEDIUM

- **[F001]** `ycc/skills/_shared/references/worktree-strategy.md:163` — Conflict-output stream documented as stdout, but `merge-children.sh:88` writes to stderr
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Update line 163 ("Prints `CONFLICT: <task-id> at <path>` to stdout.") and line 195 ("Emit `CONFLICT: <task-id> at <path>` to stdout") to say `stderr`. The calling skills already document "prints CONFLICT" without specifying stream and work correctly; only the canonical reference is wrong, so a future parser built against it would miss conflicts.

- **[F002]** `ycc/skills/_shared/scripts/merge-children.sh:80-83` — `git worktree remove` / `git branch -d` failures are swallowed with `|| true`
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: Replace `|| true` with explicit error handling. If `worktree remove` fails (e.g., uncommitted files in the child), print a `WARN: worktree remove failed for <task-id> at <path>` to stderr instead of silently continuing. Same for `branch -d`. Otherwise the next run finds stale directories and errors out of the idempotency check in `setup-worktree.sh` without explaining why. Consider `git worktree remove --force` after a successful merge, since the child is now represented in the parent branch and the worktree state no longer matters.

- **[F003]** `ycc/settings/settings.json:69` — Hook points at `~/.claude/hooks/worktree-create.sh` but that path won't resolve until the user runs the `ln -s` documented in `CONTRIBUTING.md:47`
  - **Status**: Fixed
  - **Category**: Completeness
  - **Suggested fix**: Add a preflight check somewhere obvious — either in `scripts/validate.sh`, a new `scripts/install-hooks.sh`, or a warning emitted by `setup-worktree.sh` when it detects the hook is registered but missing. Alternative: change the hook command to `bash "$(dirname "$0")/../ycc/settings/hooks/worktree-create.sh"` so it works without the symlink (but this couples the settings file to repo-root layout). Document explicitly that until the symlink is created, `Agent(isolation: "worktree")` falls back to the harness default (pollutes `<repo>/.claude/worktrees/`).

- **[F004]** `ycc/agents/implementor.md` — No explicit guidance to expand `~` in `Working directory:` paths before passing to tools
  - **Status**: Fixed
  - **Category**: Correctness
  - **Suggested fix**: The worktree plan annotations use literal `~/.claude-worktrees/...` strings. An implementor agent that copies the `~` into a `Read` or `Edit` call will fail — most tool invocations don't shell-expand `~`. Add a sentence to the "Working Directory" section: "Expand `~` to `$HOME` (or the absolute user home path) before using the path in any tool call; tool invocations do not perform shell expansion."

### LOW

- **[F005]** `ycc/settings/hooks/worktree-create.sh:45-49` — JSON fallback parser is fragile (`grep` + `sed`)
  - **Status**: Open
  - **Category**: Maintainability
  - **Suggested fix**: The fallback handles simple `"field": "value"` pairs but breaks on escaped quotes, unicode escapes, or nested objects. The Claude Code hook payload should be well-formed, so the risk is low. Prefer requiring `jq` (fail fast with a clear error if missing) or use `python3 -c 'import json, sys; print(json.load(sys.stdin).get("field", ""))'` as a more robust fallback.

- **[F006]** `ycc/settings/hooks/worktree-create.sh:60-73` — Misleading "fail open" comment; behavior actually emits a likely-invalid path
  - **Status**: Open
  - **Category**: Maintainability
  - **Suggested fix**: Comment says "fail open" but the path emitted (`"$CWD/.claude/worktrees/fallback"`) is not actually the harness default — it's a guess. Replace with either (a) `exit 1` with a clear stderr message so the harness logs the failure, or (b) emit the original `INTENDED_PATH` unchanged so the harness proceeds with whatever it was going to do.

- **[F007]** `ycc/skills/_shared/scripts/list-worktrees.sh:146` — Uses `seq` which is not POSIX
  - **Status**: Open
  - **Category**: Pattern Compliance
  - **Suggested fix**: `seq` is GNU/BSD but not POSIX. Works on Linux/macOS which covers the target platforms, so this is informational only. If cross-platform reach matters, replace with a bash-native loop: `for ((i=1; i<=max_path; i++)); do printf -; done`.

- **[F008]** `ycc/skills/_shared/scripts/merge-children.sh:99-120` — Inline `echo ERROR` + `usage` pattern duplicated 4 times
  - **Status**: Open
  - **Category**: Maintainability
  - **Suggested fix**: Extract a `die()` helper: `die() { echo "ERROR: $*" >&2; usage; }`. Cosmetic; doesn't change behavior.

- **[F009]** `ycc/skills/orchestrate/SKILL.md` and `ycc/skills/implement-plan/SKILL.md` exceed the 500-line soft cap (768 and 741 lines respectively)
  - **Status**: Open
  - **Category**: Maintainability
  - **Suggested fix**: Explicitly deferred by the user during planning ("no refactor needed right now"). Noted for follow-up. The `--worktree` additions themselves are terse; the overages pre-existed. If split is pursued later, extract the Path A/B dispatch templates into `ycc/skills/_shared/references/*.md` so the main SKILL.md files shrink.

- **[F010]** `ycc/skills/_shared/references/worktree-strategy.md:163,195` vs. `:201-202` — Mixed guidance on what the skill does after conflict
  - **Status**: Open
  - **Category**: Maintainability
  - **Suggested fix**: §5 step 2 says "Display the conflicting file(s) from the child branch for manual review" but the script already ran `git merge --abort` in the parent, leaving nothing to inspect there. Clarify the resolution workflow: user must `cd` into the child worktree (still present because `merge-children.sh` exited before cleanup), inspect, then manually `git cherry-pick` or re-merge in the parent. Or document that the user should `--abort` any state and restart the batch.

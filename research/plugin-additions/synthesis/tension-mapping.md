# Tension Mapping: plugin-additions

## Executive Summary

The highest-value tensions cluster around growth versus integrity. The repo can either keep expanding its visible catalog or invest in the invisible machinery that makes that catalog trustworthy. The research strongly favors the second path, with one exception: hook workflows now look strategically timely.

## Maximum Disagreement Points

### 1. Breadth vs integrity

- **Tension**: More visible skills/agents vs fewer sync and release failures
- **Why it matters**: This determines whether future additions compound value or compound entropy
- **Current best move**: Integrity first

### 2. Cross-platform ambition vs target-specific reality

- **Tension**: One unified workflow story vs platform-specific feature matrices
- **Why it matters**: `ycc` supports Claude, Cursor, and Codex; pretending they are identical creates broken promises
- **Current best move**: Generate per-target behavior from one shared source

### 3. Prose guidance vs executable policy

- **Tension**: Rules/hook markdown vs actual hook installation and validation
- **Why it matters**: The repo already has extensive hook guidance; user value now requires executable follow-through
- **Current best move**: Add a hook workflow skill backed by scripts

### 4. Manual curation vs generated inventory

- **Tension**: Hand-edited README counts and parity assumptions vs generated source-of-truth reporting
- **Why it matters**: Silent drift is already visible
- **Current best move**: Generate inventories

## Most Important Tension

**Prose system vs product system**  
`ycc` has matured beyond being just a collection of skill text. It now needs product-grade authoring, packaging, release, and compatibility workflows.

## Practical Implication

The next 2-4 additions should mostly be "repo-operating skills" rather than new subject-matter skills.
